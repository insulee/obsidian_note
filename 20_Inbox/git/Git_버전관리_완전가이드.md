## Git이란 무엇인가?

### 간단한 비유로 이해하기

**Git은 프로젝트의 "타임머신"이라고 생각하시면 됩니다.**

일반적인 문서 편집에서 "저장" 버튼을 누르면 이전 버전은 사라지고 새 버전만 남습니다. 하지만 Git을 사용하면:

- ✅ **모든 변경 이력이 기록됩니다** - 언제든지 이전 버전으로 돌아갈 수 있습니다
- ✅ **변경 내용을 추적합니다** - 어떤 파일을 언제 누가 왜 수정했는지 알 수 있습니다
- ✅ **여러 버전을 동시에 관리합니다** - 새로운 기능을 안전하게 테스트하면서도 안정적인 버전을 유지할 수 있습니다

### 예시: 제품 문서 관리 시스템

제품 매뉴얼과 가이드 문서를 관리할 때:
- **1주차**: 기본 문서 구조 설계 및 초안 작성
- **2주차**: 제품 기능 설명 추가
- **3주차**: 실수로 중요한 내용을 삭제했어요! 이전 버전으로 되돌려야 함

Git 없이는: 3주차 작업을 모두 취소하고 처음부터 다시 작성해야 합니다.

Git이 있으면: 2주차 버전으로 되돌아가서 삭제된 내용만 복구하면 됩니다.

## 왜 Git을 사용하는가?

### 1. 실수로 코드를 망가뜨려도 걱정 없음

```python
# 작업 중 실수로 중요한 코드를 삭제했어요
# Git이 있으면: git checkout 이전버전으로 파일 복구 가능
# Git 없으면: 백업을 찾아야 하는데... 백업이 어디 있었지? 😱
```

### 2. 여러 기능을 동시에 개발 가능

- 메인 코드: 안정적인 버전 유지
- 새 기능 브랜치: 실험적인 기능 개발
- 각각 독립적으로 작업 가능

### 3. 협업이 쉬워짐

- 여러 사람이 같은 파일을 수정해도 Git이 자동으로 합쳐줌
- 누가 무엇을 변경했는지 명확히 추적 가능

### 4. 문서 변경 이력도 관리

- 마크다운 문서도 코드와 마찬가지로 버전 관리
- "이전에 작성했던 내용이 뭐였지?" → Git 히스토리에서 확인

## Git의 핵심 개념

### 1. 저장소 (Repository, 줄여서 Repo)

**저장소 = 프로젝트 폴더 + Git 관리 시스템**

```
제품문서관리/
├── .git/          ← Git이 관리 정보를 저장하는 폴더 (숨김 폴더)
├── 매뉴얼/
│   ├── 사용자_가이드.md
│   └── 설치_매뉴얼.md
├── 문서/
│   ├── FAQ.md
│   └── API_문서.md
└── 설정파일들
```

**저장소 종류:**
- **로컬 저장소**: 내 컴퓨터에 있는 저장소
- **원격 저장소**: GitHub, GitLab 같은 온라인 저장소 (백업 + 공유)

### 2. 커밋 (Commit)

**커밋 = 변경사항의 "스냅샷"**

커밋은 마치 "체크포인트 저장" 같은 개념입니다:

```
커밋 1: "프로젝트 초기 설정 완료"
커밋 2: "사용자 가이드 초안 작성"
커밋 3: "제품 설치 매뉴얼 추가"
커밋 4: "문서 오타 수정 및 구조 개선"
```

**커밋에는 다음 정보가 포함됩니다:**
- 변경된 파일들
- 변경 내용
- 누가 변경했는지 (작성자)
- 언제 변경했는지 (날짜/시간)
- 왜 변경했는지 (커밋 메시지)

### 3. 스테이징 (Staging)

**스테이징 = "이번 커밋에 포함시킬 파일 선택"**

작업 흐름:
1. 파일 수정 → Git이 변경사항 감지
2. **스테이징** → 커밋에 포함시킬 파일 선택 (`git add`)
3. **커밋** → 선택한 파일들을 하나의 스냅샷으로 저장 (`git commit`)

**왜 스테이징이 필요한가?**
- 관련 없는 변경사항을 분리해서 커밋 가능
- 예: 매뉴얼 업데이트 + FAQ 오타 수정 → 두 개의 별도 커밋으로 관리

### 4. 브랜치 (Branch)

**브랜치 = 독립적인 작업 공간**

```
main (메인 브랜치)
  ├── [기존 안정적인 코드]
  │
  ├── feature/새기능 (새 기능 개발)
  │   └── [실험적인 코드]
  │
  └── fix/버그수정 (버그 수정)
      └── [수정된 코드]
```

**브랜치를 사용하는 이유:**
- 메인 코드는 안정적으로 유지
- 새 기능은 별도 브랜치에서 안전하게 테스트
- 테스트 완료 후 메인에 병합

### 5. 푸시 (Push)와 풀 (Pull)

**푸시 (Push)**: 로컬 저장소의 변경사항을 원격 저장소에 업로드
```
내 컴퓨터 → GitHub (백업 + 공유)
```

**풀 (Pull)**: 원격 저장소의 변경사항을 로컬 저장소로 다운로드
```
GitHub → 내 컴퓨터 (최신 버전 가져오기)
```

### 6. 원격 저장소 (Remote Repository)

**원격 저장소 = 온라인 백업 + 협업 공간**

주요 서비스:
- **GitHub**: 가장 인기 있는 서비스 (무료)
- **GitLab**: 오픈소스 친화적
- **Bitbucket**: Atlassian 제품군과 통합

**장점:**
- ✅ 코드 백업 (컴퓨터 고장 시에도 안전)
- ✅ 어디서나 접근 가능
- ✅ 다른 사람과 공유 및 협업

## 일상적인 작업 흐름

### 기본 워크플로우

```
1. 파일 수정
   ↓
2. 변경사항 확인 (git status)
   ↓
3. 스테이징 (git add)
   ↓
4. 커밋 (git commit)
   ↓
5. 푸시 (git push) - 원격 저장소에 업로드
```

### 실제 예시: 문서 수정하기

#### 예시 1: 기존 문서 수정

**1단계: 파일 수정**
```markdown
# 사용자_가이드.md 파일을 수정했습니다
# 이전 내용: "제품을 설치하세요"
# 새 내용: "제품 설치 방법을 아래 단계별로 따라하세요"
```

**2단계: 변경사항 확인**
```bash
git status
```
출력 예시:
```
변경된 파일:
  사용자_가이드.md (수정됨)
```

**3단계: 스테이징**
```bash
git add 사용자_가이드.md
# 또는 모든 변경사항: git add .
```

**4단계: 커밋**
```bash
git commit -m "docs: 사용자 가이드 설치 섹션 개선"
```

**5단계: 푸시 (원격 저장소가 있다면)**
```bash
git push
```

#### 예시 2: 새로 만든 문서 추가하기

**1단계: 새 문서 작성**
```markdown
# 새 문서 "고급_기능_가이드.md"를 만들었어요
```

**2단계: 변경사항 확인**
```bash
git status
```
출력 예시:
```
Untracked files:
  (use "git add <file>..." to include in what will be committed)
        고급_기능_가이드.md
```

**3단계: 스테이징 (수정된 파일과 동일하게!)**
```bash
git add 고급_기능_가이드.md
# 또는 모든 변경사항: git add .
```

**4단계: 커밋**
```bash
git commit -m "docs: 고급 기능 가이드 문서 추가"
```

**5단계: 푸시**
```bash
git push
```

**핵심 포인트:**
- 수정된 문서든 새로 만든 문서든 **처리 방법이 동일합니다**
- `git add`로 스테이징하고 `git commit`으로 커밋하면 됩니다
- Git은 자동으로 파일이 새 파일인지 수정된 파일인지 구분합니다

## 브랜치 전략

### 기본 브랜치 구조

```
main (메인 브랜치)
  ├── 안정적인 버전 유지
  │
  ├── feature/기능명 (새 기능 개발)
  │   └── 예: feature/kakao-api-integration
  │
  ├── fix/버그명 (버그 수정)
  │   └── 예: fix/login-error
  │
  └── docs/문서명 (문서 작업)
      └── 예: docs/api-documentation
```

### 브랜치 작업 흐름

#### 1. 새 기능 브랜치 생성
```bash
# 메인 브랜치에서 시작
git checkout main
git pull  # 최신 버전 가져오기

# 새 브랜치 생성 및 전환
git checkout -b feature/new-chatbot-skill
```

#### 2. 브랜치에서 작업
```bash
# 코드 수정
# 문서 수정
git add .
git commit -m "새 챗봇 스킬 기능 추가"
git push origin feature/new-chatbot-skill
```

#### 3. 작업 완료 후 메인에 병합
```bash
# 메인 브랜치로 돌아가기
git checkout main

# 새 기능 브랜치 병합
git merge feature/new-chatbot-skill

# 원격 저장소에 반영
git push
```

### 권장 브랜치 명명 규칙

- `feature/기능명`: 새 기능 개발
- `fix/버그명`: 버그 수정
- `docs/문서명`: 문서 작업
- `refactor/내용`: 코드 리팩토링

예시:
- `feature/advanced-search`
- `fix/installation-guide-error`
- `docs/api-guide-update`

## 마크다운 문서와 코드 함께 관리하기

### 마크다운 문서도 버전관리가 필요한 이유

1. **문서 변경 이력 추적**
   - "이전에 작성했던 내용이 뭐였지?"
   - "언제 이 가이드를 작성했지?"
   - "이 문서를 누가 수정했지?"

2. **문서 리뷰 및 협업**
   - 코드 리뷰처럼 문서도 리뷰 가능
   - 여러 사람이 문서를 수정해도 Git이 자동으로 합침

3. **문서와 코드의 일관성 유지**
   - 코드 변경 시 관련 문서도 함께 업데이트
   - 같은 커밋에 포함하여 일관성 보장

### 실전 예시: 제품 기능 문서 추가하기

**시나리오**: 새 제품 기능을 추가하면서 관련 문서도 작성

```bash
# 1. 새 브랜치 생성
git checkout -b docs/new-feature-guide

# 2. 새 문서 작성
# - 매뉴얼/새기능_가이드.md 파일 작성
# - FAQ.md에 새 기능 관련 질문 추가

# 3. 기존 문서 업데이트
# - 사용자_가이드.md에 새 기능 섹션 추가
# - 설치_매뉴얼.md 업데이트

# 4. 모든 변경사항을 하나의 커밋으로
git add .
git commit -m "docs: 새 기능 가이드 문서 추가 및 관련 문서 업데이트"

# 5. 푸시
git push origin docs/new-feature-guide
```

**핵심 포인트:**
- 새 문서와 수정된 문서를 **같은 커밋**에 포함하면 일관성 유지
- 관련 변경사항을 **논리적으로 그룹화**하여 커밋
- 새로 만든 문서도 `git add`로 추가하면 됩니다

### 옵시디언 문서 관리 팁

1. **.obsidian 폴더는 제외**
   - `.gitignore`에 `.obsidian/` 추가
   - 개인 설정 파일은 버전관리 불필요

2. **대용량 첨부파일 관리**
   - 이미지, PDF 등은 Git LFS 사용 또는 별도 저장소
   - 또는 문서만 관리하고 첨부파일은 별도 관리

3. **문서 구조 일관성**
   - 일관된 폴더 구조 유지
   - 파일명 규칙 준수 (예: `YYMMDD-문서명.md`)

## 요약

- **Git** = 프로젝트의 타임머신 (버전 관리 시스템)
- **저장소** = Git이 관리하는 프로젝트 폴더
- **커밋** = 변경사항의 스냅샷
- **브랜치** = 독립적인 작업 공간
- **스테이징** = 커밋에 포함시킬 파일 선택

코드와 마크다운 문서를 구분할 필요 없이 **모두 Git으로 관리**하면 됩니다!
