## 마크다운 문서도 버전관리가 필요한 이유

### 1. 문서 변경 이력 추적

**문제 상황:**
- "이전에 작성했던 내용이 뭐였지?"
- "언제 이 가이드를 작성했지?"
- "이 문서를 누가 수정했지?"
- "실수로 중요한 내용을 삭제했어요!"

**Git의 해결책:**
- 모든 변경 이력이 기록됨
- 언제든지 이전 버전으로 되돌릴 수 있음
- 변경 내용을 한눈에 확인 가능

### 2. 문서와 코드의 일관성 유지

**제품 문서 관리 시스템 예시:**

```
상황: 새 제품 기능을 추가했습니다
- 문서: 사용자_가이드.md 업데이트
- 문서: 설치_매뉴얼.md 업데이트 필요

Git 없이: 관련 문서를 업데이트하는 것을 깜빡할 수 있음
Git 사용: 같은 커밋에 포함하여 일관성 보장
```

**실제 예시:**
```bash
# 관련 문서들을 함께 커밋
git add "사용자_가이드.md" "설치_매뉴얼.md"
git commit -m "docs: 새 기능 관련 문서 업데이트"
```

### 3. 문서 리뷰 및 협업

- 코드 리뷰처럼 문서도 리뷰 가능
- 여러 사람이 문서를 수정해도 Git이 자동으로 합침
- 누가 무엇을 변경했는지 명확히 추적

### 4. 문서 구조 개선 이력 관리

```
2025-01-15: 시나리오 설계 문서 초안 작성
2025-01-20: 사용자 피드백 반영하여 구조 개선
2025-01-25: 새 시나리오 추가 및 예시 보강
```

Git으로 이런 모든 변경 이력을 추적할 수 있습니다.

## 옵시디언과 Git 연동

### 옵시디언 폴더 구조

```
제품문서관리/
├── .obsidian/              ← 옵시디언 설정 (Git 제외)
│   ├── config.json
│   ├── plugins/
│   └── ...
├── .git/                    ← Git 저장소
├── .gitignore               ← Git 제외 파일 목록
├── 매뉴얼/                  ← 마크다운 문서 (Git 관리)
│   ├── 사용자_가이드.md
│   └── 설치_매뉴얼.md
├── 문서/                    ← 마크다운 문서 (Git 관리)
│   ├── FAQ.md
│   └── API_문서.md
└── ...
```

### .gitignore 설정

옵시디언 관련 파일 중 Git에 포함시키지 않을 것들:

```gitignore
# 옵시디언 설정 폴더 (개인 설정이므로 제외)
.obsidian/

# 옵시디언 캐시
.obsidian/workspace.json
.obsidian/workspace-mobile.json

# 옵시디언 플러그인 데이터
.obsidian/plugins/*/data.json
```

**포함할 것:**
- ✅ 마크다운 문서 파일 (`.md`)
- ✅ 이미지 파일 (`.png`, `.jpg` 등) - 필요시
- ✅ PDF 파일 - 필요시

**제외할 것:**
- ❌ `.obsidian/` 폴더 (개인 설정)
- ❌ 임시 파일
- ❌ 대용량 바이너리 파일 (선택사항)

### 옵시디언에서 Git 사용하기

옵시디언은 Git과 직접 통합되지 않지만, 터미널이나 Git GUI 도구를 사용할 수 있습니다.

**방법 1: 터미널 사용**
```bash
cd "프로젝트 폴더 경로"
git add .
git commit -m "문서 업데이트"
git push
```

**방법 2: Git GUI 도구 사용**
- **SourceTree**: 무료, 사용하기 쉬움
- **GitKraken**: 고급 기능 제공
- **VS Code**: 내장 Git 기능

**방법 3: 옵시디언 플러그인 (선택사항)**
- `Git` 플러그인: 옵시디언 내에서 Git 명령 실행

## 문서 변경사항 추적

### 문서 변경 이력 확인

```bash
# 특정 문서의 변경 이력
git log "5_Documents/시나리오_설계.md"

# 변경 내용까지 확인
git log -p "5_Documents/시나리오_설계.md"

# 간단한 변경 이력
git log --oneline "5_Documents/시나리오_설계.md"
```

출력 예시:
```
abc123 docs: 시나리오 설계 문서에 새 플로우 추가
def456 docs: 시나리오 설계 문서 초안 작성
```

### 문서 변경 내용 비교

```bash
# 두 커밋 간의 차이
git diff 커밋1 커밋2 -- "5_Documents/시나리오_설계.md"

# 현재 작업 중인 변경사항
git diff "5_Documents/시나리오_설계.md"
```

### 문서의 특정 버전 보기

```bash
# 특정 커밋 시점의 문서 내용
git show 커밋해시:"5_Documents/시나리오_설계.md"

# 이전 버전으로 되돌리기
git checkout 커밋해시 -- "5_Documents/시나리오_설계.md"
```

### 문서 변경 통계

```bash
# 문서의 변경 통계
git log --stat "5_Documents/시나리오_설계.md"

# 작성자별 기여도
git log --pretty=format:"%an" -- "5_Documents/시나리오_설계.md" | sort | uniq -c
```

## 문서 리뷰 프로세스

### 브랜치 기반 문서 리뷰

코드 리뷰처럼 문서도 브랜치를 사용하여 리뷰할 수 있습니다.

#### 1단계: 문서 작업 브랜치 생성

```bash
git checkout -b docs/api-guide-update
```

#### 2단계: 문서 수정

```bash
# API_가이드.md 파일 수정
git add "5_Documents/API_가이드.md"
git commit -m "docs: API 가이드 문서 전면 개정"
```

#### 3단계: 원격 저장소에 푸시

```bash
git push -u origin docs/api-guide-update
```

#### 4단계: 리뷰 요청

GitHub/GitLab에서 Pull Request 생성:
- 변경 내용 확인
- 리뷰어 지정
- 피드백 반영

#### 5단계: 리뷰 후 병합

```bash
# 피드백 반영 후
git add "5_Documents/API_가이드.md"
git commit -m "docs: 리뷰 피드백 반영"
git push

# 메인 브랜치에 병합
git checkout main
git merge docs/api-guide-update
git push
```

### 문서 리뷰 체크리스트

- [ ] 내용의 정확성
- [ ] 문서 구조의 명확성
- [ ] 예시의 적절성
- [ ] 오타 및 문법 오류
- [ ] 링크 및 참조의 유효성
- [ ] 코드와 문서의 일관성

#### 상황: API 기능 추가 및 문서 업데이트

```bash
# 1. 새 브랜치 생성
git checkout -b docs/new-feature-guide

# 2. 새 문서 작성
# - 매뉴얼/새기능_가이드.md 파일 작성
# - 문서/FAQ.md에 새 기능 관련 질문 추가

# 3. 기존 문서 업데이트
# - 매뉴얼/사용자_가이드.md에 새 기능 섹션 추가
# - 매뉴얼/설치_매뉴얼.md 업데이트

# 4. 모든 변경사항을 하나의 커밋으로
git add "매뉴얼/새기능_가이드.md" "문서/FAQ.md" "매뉴얼/사용자_가이드.md" "매뉴얼/설치_매뉴얼.md"
git commit -m "docs: 새 기능 가이드 문서 추가 및 관련 문서 업데이트

- 새 기능 가이드 문서 작성
- FAQ에 새 기능 관련 질문 추가
- 사용자 가이드에 새 기능 섹션 추가
- 설치 매뉴얼 업데이트"

# 5. 푸시
git push -u origin docs/new-feature-guide
```

**장점:**
- 코드와 문서가 같은 커밋에 포함되어 일관성 유지
- 변경 이력을 한눈에 확인 가능

## 대용량 첨부파일 관리

### 문제: Git은 대용량 파일에 적합하지 않음

Git은 텍스트 파일(코드, 마크다운)에 최적화되어 있습니다. 대용량 바이너리 파일은:
- 저장소 크기가 커짐
- 클론 시간이 오래 걸림
- 성능 저하

### 해결 방법 1: .gitignore로 제외

**작은 이미지나 PDF는 포함 가능:**
```gitignore
# 대용량 파일 제외 (선택사항)
*.mp4
*.mov
*.zip
*.rar

# 특정 크기 이상의 파일 제외는 자동으로 불가능
# 수동으로 .gitignore에 추가 필요
```

**실제 예시:**
```gitignore
# 비디오 파일 제외
**/*.mp4
**/*.mov

# 큰 이미지 파일 제외 (선택사항)
**/*.psd
**/*.ai
```

### 해결 방법 2: Git LFS (Large File Storage)

**대용량 파일을 별도로 관리:**

```bash
# Git LFS 설치 후
git lfs install

# 추적할 파일 타입 지정
git lfs track "*.mp4"
git lfs track "*.pdf"

# .gitattributes 파일 커밋
git add .gitattributes
git commit -m "chore: Git LFS 설정"
```

**장점:**
- 대용량 파일도 버전 관리 가능
- 저장소 크기 최적화

**단점:**
- GitHub 무료 계정은 1GB 제한
- 설정이 복잡함

### 해결 방법 3: 별도 저장소 또는 클라우드

**추천 방법:**
- 문서는 Git으로 관리
- 대용량 첨부파일은 별도 클라우드 저장소 사용
- 문서에 링크로 참조

**예시:**
```markdown
# 사용자 가이드

[제품 구조 다이어그램](https://cloud.example.com/files/product-structure.png)
```

### 권장 사항

**제품 문서 관리 시스템:**
- ✅ 마크다운 문서: Git으로 관리
- ✅ 작은 이미지 (PNG, JPG < 1MB): Git으로 관리 가능
- ✅ PDF 문서: Git으로 관리 가능 (작은 경우)
- ❌ 비디오 파일: Git 제외 또는 Git LFS
- ❌ 대용량 이미지: Git 제외 또는 Git LFS

## 옵시디언 문서 관리 모범 사례

### 1. 파일명 규칙

**일관된 파일명 규칙 사용:**

```
✅ 좋은 예시:
- 2025-01-21-사용자_가이드.md
- API_가이드.md
- 제품_문서_관리_시스템_개요.md

❌ 나쁜 예시:
- 가이드.md (날짜 없음)
- api guide.md (공백, 소문자)
- 새문서.md (의미 불명확)
```

### 2. 폴더 구조

**논리적인 폴더 구조:**

```
매뉴얼/
├── 사용자_가이드.md
├── 설치_매뉴얼.md
└── 고급_기능_가이드.md

문서/
├── FAQ.md
├── API_문서.md
└── 문제해결_가이드.md
```

### 3. 문서 메타데이터

**프론트매터(Frontmatter) 사용:**

```markdown
---
Created_date: 2025-01-21
Last_Modified_date: 2025-01-21
Version: 1.0
Status: Active
Category: guide
tags:
  - guide
  - api
  - documentation
---

# 문서 제목
```

**Git으로 버전 관리하면 `Last_Modified_date`는 Git 히스토리에서 확인 가능합니다.**

### 4. 문서 간 링크

**옵시디언 내부 링크:**

```markdown
[[시나리오_설계]] 문서를 참고하세요.
```

**Git으로 관리하면 링크가 깨지지 않도록 주의:**
- 파일명 변경 시 모든 참조 업데이트
- Git 히스토리에서 링크 변경 이력 추적 가능

### 5. 문서 버전 관리 전략

#### 전략 A: 주요 문서만 버전 관리

```gitignore
# 일일 업무 노트는 Git 제외
22_Area/20_업무노트/

# 프로젝트 문서만 Git 관리
21_Project/**/*.md
```

#### 전략 B: 모든 문서 버전 관리

```bash
# 모든 마크다운 파일 관리
git add **/*.md
```

**제품 문서 관리 시스템 권장:**
- ✅ 프로젝트 문서: 버전 관리
- ✅ 공유 문서: 버전 관리
- ✅ 새로 만든 문서: 버전 관리 (처리 방법 동일)
- ❌ 개인 업무 노트: 선택사항

### 6. 문서 변경 커밋 전략

#### 작은 단위로 자주 커밋

```bash
# ✅ 좋은 예시: 논리적인 단위로 커밋
git commit -m "docs: 사용자 가이드 - 설치 섹션 추가"
git commit -m "docs: 사용자 가이드 - 문제해결 섹션 추가"

# ❌ 나쁜 예시: 모든 변경사항을 한 번에
git commit -m "문서 업데이트"
```

## 실제 워크플로우 예시

### 시나리오: 제품 설치 가이드 문서 작성

```bash
# 1. 새 브랜치 생성
git checkout -b docs/installation-guide

# 2. 새 문서 작성
# - 설치_매뉴얼.md 파일 작성
# - 단계별 스크린샷 이미지 추가
# - 문제해결 섹션 추가

# 3. 관련 문서도 함께 업데이트
# - 사용자_가이드.md에 설치 링크 추가
# - FAQ.md에 설치 관련 질문 추가

# 4. 스테이징
# 새로 만든 문서와 수정된 문서 모두 포함
git add "매뉴얼/설치_매뉴얼.md"
git add "매뉴얼/사용자_가이드.md"
git add "문서/FAQ.md"
git add "매뉴얼/attachment/installation-screenshot.png"

# 5. 커밋
git commit -m "docs: 설치 매뉴얼 문서 작성

- 설치 단계별 가이드 작성
- 스크린샷 이미지 추가
- 사용자 가이드에 설치 링크 추가
- FAQ에 설치 관련 질문 추가"

# 6. 푸시
git push -u origin docs/installation-guide

# 7. 리뷰 후 병합 (GitHub/GitLab에서)
git checkout main
git merge docs/installation-guide
git push
```

**중요**: 새로 만든 문서(`설치_매뉴얼.md`)든 수정된 문서(`사용자_가이드.md`)든 모두 `git add`로 스테이징하면 됩니다!
